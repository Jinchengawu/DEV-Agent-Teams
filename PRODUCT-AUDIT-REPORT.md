## 二、功能短板分析（按严重度分级）

### 🔴 P0 — 致命短板（没有这些，产品无法闭环）

#### P0-1：Agent 输出无法执行（代码是"死的"）

**问题**：Frontend Agent 写了一个 React 组件，你只能看到代码文本，不能：
- 在浏览器中预览效果
- 运行单元测试验证
- 部署到临时环境

**影响**：这是**开发工具**和**聊天工具**的本质区别。当前 DEV-Agent-Teams 更像一个"AI 聊天室"，而不是"AI 开发环境"。

**用户原话模拟**：
> "Agent 帮我写了个登录页面，然后呢？我 copy-paste 到本地项目里再自己运行？那我要你这 6 个 Agent 干嘛？"

**竞品对比**：
| 产品 | 代码执行能力 |
|------|-------------|
| Bolt.new / V0.dev | 云端实时预览、自动部署 |
| Replit Agent | 完整运行环境、依赖自动安装 |
| GitHub Copilot Workspace | 代码生成 + 直接提交 PR |
| **DEV-Agent-Teams** | ❌ 纯文本输出，无执行环境 |

#### P0-2：无文件/多模态输入能力

**问题**：用户只能输入文本。不能：
- 上传设计稿（.png/.fig）让 Frontend Agent 还原
- 上传数据库 schema（.sql）让 Backend Agent 生成 API
- 上传 PDF 需求文档让 PM Agent 分析
- 上传截图让 Agent 定位 Bug

**影响**：严重限制输入效率。现实开发中，大量上下文以非文本形式存在。

#### P0-3：无 Git 版本控制集成

**问题**：6 个 Agent 协作产出的代码，没有：
- 分支管理
- 版本历史
- 冲突解决
- PR 审查流程

**影响**：Agent 产出的代码无法纳入正常开发工作流。如果 5 个 Agent 同时修改同一个文件，结果不可控。

#### P0-4：无用户/权限/多租户系统

**问题**：当前没有任何用户概念：
- 没有登录/注册
- 没有项目隔离（任何人都能看到所有会话）
- 没有权限控制（谁可以执行 Meeting？谁可以删除会话？）
- 没有 API Key 管理

**影响**：无法团队协作，无法商业化，数据安全性为零。

---

### 🟠 P1 — 严重短板（有替代方案，但体验极差）

#### P1-1：无知识库 / 长期记忆 / RAG

**问题**：每次新会话都是"冷启动"：
- Agent 不记得上次的项目架构决策
- 不记得代码规范
- 不记得用户偏好
- 没有项目文档索引

**现状**：SQLite 只存了 `messages`（对话记录），但对话历史是**线性文本**，不是**结构化知识**。

**影响**：Agent 无法做"持续迭代式开发"，只能做"一次性任务"。

#### P1-2：无实时通知 / 事件驱动机制

**问题**：当 Team 模式完成时：
- 用户不会收到通知（除非一直盯着 Dashboard）
- 任务阻塞时无人知晓
- Agent 异常时无告警

**现状**：只有 Dashboard 的 `/health` 轮询，没有主动推送。

#### P1-3：无 Token / 成本监控

**问题**：用户不知道：
- 本月花了多少 Token
- 哪个 Agent 最耗资源
- 单次 Team 模式成本
- 预算预警

**现状**：数据库有 `tokens` 字段，但 Dashboard 没有可视化。

#### P1-4：无代码沙箱 / 安全执行环境

**问题**：即使未来加了代码执行，也需要：
- 隔离环境（Docker/container）
- 超时限制（防止无限循环）
- 资源限制（CPU/内存上限）
- 安全扫描（防止恶意代码）

**现状**：完全空白。

#### P1-5：Testing Agent 不能实际运行测试

**问题**：Testing Agent 能"写测试代码"，但不能：
- 运行 `pytest` 或 `jest`
- 生成覆盖率报告
- 在 CI 环境中执行
- 回归验证

**本质**：Testing Agent 是"测试代码写手"，不是"测试执行器"。

#### P1-6：DevOps Agent 不能实际部署

**问题**：DevOps Agent 能"写 Dockerfile"，但不能：
- 构建 Docker 镜像
- 推送到 Registry
- 部署到 K8s
- 执行 Helm 升级
- 查看 Pod 日志

**本质**：DevOps Agent 是"配置写手"，不是"运维执行器"。

---

### 🟡 P2 — 体验短板（有则更好，无则可用）

#### P2-1：无会话搜索 / 语义检索

- 118 个会话，只能分页浏览，不能搜索
- 不能按关键词、Agent、时间范围筛选
- 没有语义搜索（"找一下上次做的登录模块"）

#### P2-2：无输出物管理 / 代码仓库

- Agent 产出的代码片段没有保存为文件
- 没有"项目"概念，会话之间没有关联
- 不能导出会话为 Markdown/PDF
- 不能生成项目报告

#### P2-3：无反馈 / 质量评分机制

- 用户无法给 Agent 输出打分
- 没有"好/坏"反馈来训练 Agent
- 没有 A/B 对比（同一任务不同 Agent 的输出对比）

#### P2-4：无移动端 / 响应式优化

- Dashboard 在手机上不可用
- 无法随时随地查看进度

#### P2-5：无模板 / 市场

- 不能复用之前的会话配置
- 不能保存"团队模板"分享给他人
- 没有预设的 Prompt 模板

## 五、关键产品决策建议

### 决策1：是否做"代码执行沙箱"？

**建议**：**必须做，优先级 P0**。

不做沙箱，DEV-Agent-Teams 永远是一个"对话工具"，不是"开发工具"。建议方案：
- **V1**：Docker Compose 启动一个 Node.js 容器，Gateway 通过 `docker exec` 执行代码
- **V2**：WebContainer（类似 StackBlitz）在浏览器中运行
- **V3**：云端沙箱（Firecracker microVM）

### 决策2：是否接入第三方 Git 平台？

**建议**：**先做 Git 本地集成，再考虑第三方**。

- V1：在 SQLite 中维护"代码快照"（每个会话的产出保存为 git-like 版本）
- V2：对接 GitHub API，自动创建分支/PR
- V3：对接 GitLab / Gitee

### 决策3：是否自建用户系统？

**建议**：**V1 用 Clerk/Auth0 快速接入，不要自研**。

- 自研用户系统（注册/登录/密码重置/邮件验证）至少 2 周
- Clerk 接入只需要 1 天，支持 OAuth（Google/GitHub/微信）
- 成本：免费额度够前期使用

### 决策4：知识库怎么做？

**建议**：**V1 用 SQLite + 向量扩展（sqlite-vec），V2 再考虑 Pinecone/Milvus**。

- 在 `messages` 表中增加向量索引
- 用 OpenAI/MiMo 的 Embedding API 生成向量
- 检索时做相似度搜索，作为 RAG 上下文注入

## 六、一句话总结

> **DEV-Agent-Teams 当前是一个"能对话的 AI 开发团队"，但不是一个"能交付的 AI 开发平台"。**  
> 从"聊天"到"交付"的鸿沟，需要**代码执行、文件处理、Git 集成、用户系统**来填补。
